# M7: v0.1 仕上げ

親プラン: [2026-07-15-v0.1-implementation.md](2026-07-15-v0.1-implementation.md)

## ゴール

v0.1.0の完了条件を満たす: HostExecutor（ホストネイティブビルド）、コンテナイメージ出力、v1.16.3/v1.15.1スキーマ、ドキュメント整備。

## 成果物

1. **HostExecutor**（`backend/internal/executor/host.go`）
   - プリフライト: ホストの `go` が `versions.json` の `goVersion` 以上（GOTOOLCHAIN autoによる自動取得も可とし、goコマンドの存在と実行可否を確認）、`node`/`npm` の存在。不足時は明確なエラーメッセージ（何をインストールすべきか）
   - `make alloy` をワークスペースで直接実行。`GOOS`/`GOARCH` はターゲット指定（デフォルト: ホストと同じ）。macOS上ならdarwinバイナリが出る。Windowsは経路を塞がないが未保証
   - このLinuxホストでは「host戦略でlinux/amd64バイナリが作れる」ことを検証（darwinの実機確認はユーザーのMacで行うため、README にデモ手順を書く）
   - 注意: ホストの `/tmp` が小さい環境向けに、ビルド時 `TMPDIR`/`GOTMPDIR` をワークスペース配下に向ける
2. **コンテナイメージ出力**（`output: "image"`）
   - DockerExecutor拡張: 改変済みワークスペースで上流Dockerfileを `docker buildx build`。単一アーチ→ `--load` してローカルDockerへ（成果物はイメージタグ名をUI表示）、複数アーチ→ `-o type=oci,dest=<name>.tar` で成果物ダウンロード
   - イメージタグ: `alloy-custom:<version>-<jobID>`
   - UIのoutput選択（binary / image）。host戦略ではimage不可（バリデーション）
3. **v1.16.3 / v1.15.1 スキーマ**
   - `./generator/run-generator.sh v1.16.3` と `v1.15.1` を実行しコミット（ジェネレータが旧バージョンで動かない場合は最小修正。`alloy` タグは1.15以降同形式のはず）
   - `schemas.yml` のmatrixに2バージョン追加。`smoke-build.yml` はv1.17.1のみのまま
   - ci.ymlのgolden validateジョブを3バージョンmatrix化（既存フィクスチャが旧バージョンでもvalidateを通るか確認。通らないフィクスチャがあればバージョン条件を付けるか設定を調整）
   - フロントエンドのバージョン選択UIが3バージョンを正しく扱えることを確認（スキーマ遅延ロードのバージョン切替）
4. **ドキュメント**
   - README全面更新: 機能概要、Pages版と ローカル版の使い分け、クイックスタート（`make frontend-build backend-build && ./backend/bin/custom-alloy-builder`）、必要環境（Docker、ディスク10GB以上、host戦略はGo/Node）、セキュリティ注記（認証なし・127.0.0.1バインド・ローカルツール）、アーキテクチャ概略（schemas生成→UI→IR→serialize、build pipeline）、macOSでのdarwinビルドデモ手順
   - `docs/decisions/` に v0.1完了時点の設計サマリー追記は不要（既存プランで足りる）

## 受け入れ条件

1. host戦略: このLinuxホストで `POST /builds {strategy: "host", targets: [{os:"linux",arch:"amd64"}]}` が成功し、成果物が `--version` に応答
2. image出力: docker戦略で `output: "image"` のジョブが成功し、`docker run --rm alloy-custom:<tag> --version` が動く
3. `schemas/v1.16.3/` と `schemas/v1.15.1/` がコミットされ、再生成diffゼロ。3バージョンがUIのバージョン選択に出る
4. golden validateが3バージョンmatrixでgreen（ローカルで3バイナリに対して実証）
5. backend/frontend全ゲート、全ワークフローgreen
6. READMEが上記内容を網羅

## 注意

- v0.1.0タグはこのタスクでは打たない（全green確認後に別途）
- ディスク残量に注意（イメージ出力検証後は `docker rmi` で掃除。生成した巨大成果物は検証後削除）
- `/tmp` は小さいtmpfs
