# custom-alloy-builder v0.1 実装計画

## Context

Grafana Alloyには「必要なコンポーネントだけを含むカスタムバイナリを作る手段」（ocb相当）と「パイプラインをグラフィカルに組んで設定を生成する手段」がない。本プロジェクトは両方を1つのWeb UIで提供する。設定ビルダー部分はGitHub Pages単体で動作し、バイナリビルドはローカルGoバックエンドで行う。要件は2026-07-15のインタビューで確定済み（`~/.claude/projects/-home-ymotongpoo-repos/memory/project_custom_alloy_builder.md` 参照）。

調査で確定した前提:
- Alloyのコンポーネント登録は `internal/component/all/all.go`（約190行の純粋なblank importリスト）。`internal/` のため外部importは不可 → クローンしたソースツリーの書き換え方式
- 構造体タグは `alloy:"name,attr|block[,optional]"`。メタデータの機械可読形式は上流に存在しない
- スキーマ生成はAST解析ではなく**リフレクション方式**: ジェネレータのGoソースをクローン内にコピーし（同一モジュール内なので `internal/` にアクセス可）、`internal/component/all` をimportしてレジストリを走査
- `make alloy` は `generate-ui`（Node/npm）に依存、`CGO_ENABLED=1` デフォルト。公式ビルドイメージ `grafana/alloy-build-image` はLinuxマルチアーチのみ（darwinクロスコンパイル不可）→ macOSはホストネイティブビルド
- alloy-configurator（Apache-2.0）の `src/lib/river.ts` にRiverシリアライザ＋tree-sitterパーサあり。シリアライザは翻案して再利用（要attribution）、パーサ（WASM）は将来のインポート機能の候補

## スキーマ形式

1コンポーネント1JSONファイル＋バージョンごとの `index.json`（palette・接続型付け・ビルド検証用の軽量サマリー: name, stability, importPath, inputs/outputs capsule ID）。UIはノード追加時に個別スキーマを遅延ロード。

型は判別共用体: `string | number | bool | duration | secret | optional_secret | map | list | object | enum | capsule | raw`。`raw` はマップ不能な型のフォールバック（UIは生テキスト入力、IRはそのまま保持）。

capsule ID: ジェネレータ内の小さなエイリアス表で正規化（`storage.Appendable → prometheus.Appendable`、`loki.LogsReceiver`、`otelcol.Consumer` はtraces/metrics/logsに分割、等）。未登録capsuleはGo型文字列をIDとしてそのまま使う。

## ジェネレータ（リフレクション）

- クローン内で `internal/component/all` をimport → 全Register発火 → レジストリ走査
- `importPath` は `runtime.FuncForPC(reflect.ValueOf(reg.Build).Pointer()).Name()` から取得（1パッケージ複数コンポーネント対応）
- `reflect.TypeOf(reg.Args/Exports)` を `alloy` タグ駆動で再帰走査。`time.Duration`→duration、`alloytypes.Secret`→secret 等。循環はvisited setで検出し `raw` で打ち切り
- Argsが `syntax.Defaulter` 実装ならインスタンス化して `SetToDefault()` を呼びスカラーデフォルトを記録
- **1フィールドの失敗で全体を落とさない**（rawへフォールバック）

## IR＋シリアライザ（TypeScript）

```ts
type IRValue = {t:"string",v} | {t:"number",v} | {t:"bool",v} | {t:"list",v}
             | {t:"map",v} | {t:"ref",target} | {t:"raw",v}   // rawがエスケープハッチ
interface IRComponent { id; type; label; body: {attrs, blocks} }
interface IRConfig { formatVersion:1; alloyVersion; components; rawSnippets[] }
```

- IRが唯一の真実。React Flowのノード/エッジはIRから導出（エッジ＝`ref` 値のスキャン）。接続可否はcapsule ID互換表で判定
- 保存形式: `{formatVersion, ir, layout}`（座標はIRの外）
- シリアライザは alloy-configurator の river.ts を翻案（Apache-2.0 attribution必須）
- 将来のインポート用に `interface Importer { parse(text): IRConfig }` だけ定義（v0.1は実装なし）

## バックエンドAPI

```
GET  /api/v1/versions
GET  /api/v1/versions/{v}/components
POST /api/v1/builds          {version, components[], targets[], output: binary|image, strategy: docker|host}
GET  /api/v1/builds/{id}     状態: queued|cloning|generating|building|done|error
GET  /api/v1/builds/{id}/logs        # SSE
GET  /api/v1/builds/{id}/artifacts/{name}
```

`Executor` インターフェース `{Build(ctx, Spec, io.Writer) ([]Artifact, error)}` に `DockerExecutor` / `HostExecutor`。直列ジョブキュー、インメモリジョブストア（ローカルツールなので永続化なし）。Pages上ではAPI疎通失敗を検知してビルダータブを「ローカル実行の案内」表示に切替。

## ビルドオーケストレーション

- クローンキャッシュ: `~/.cache/custom-alloy-builder/src/alloy-<version>`（`git clone --depth 1 --branch vX`）。ビルドごとにrsyncコピーしてから `all.go` を改変
- `all.go` 再生成: 選択コンポーネントのimportPathから生成（パッケージ単位でdedup）。他所からimportされているパッケージは除去禁止 → バージョンごとに検出して `versions.json` の `alwaysIncludePkgs` に記録
- **Docker戦略**（デフォルト）: `grafana/alloy-build-image:<versions.jsonでピン>` 内で `make alloy`（GOOS/GOARCH: linux/amd64, arm64）。イメージ出力は上流Dockerfileで `docker buildx build`（単アーチ `--load`、マルチアーチ `-o type=oci,dest=*.tar`）
- **ホスト戦略**: ローカルGo（`versions.json` のgoVersion以上）＋Nodeをプリフライトチェック → `make alloy`（macOS→darwin。Windowsは経路を塞がないが未保証・イメージ出力なし）
- ビルドコマンドの複製はせず `make alloy` を呼ぶ（`generate-ui`/`generate-source-code` の追従を上流に任せる）
- 成果物: `alloy-custom-<version>-<os>-<arch>` ＋ 選択コンポーネント一覧と `all.go` を記録した `manifest.json`

## CI（GitHub Actions）

1. **ci.yml**: フロントlint+vitest+build / `go vet`+`go test` / **goldenテスト**: `testdata/golden/` のIRをシリアライズ → 公式Alloyリリースバイナリ（3バージョンmatrix）で `alloy validate` → コミット済み.alloyとdiff
2. **schemas.yml**: ジェネレータ再実行 → `git diff --exit-code schemas/`（再現性検証）
3. **pages.yml**: フロント静的ビルド（base=`/custom-alloy-builder/`、schemas同梱）→ Pages deploy
4. **smoke-build.yml**（nightly+手動）: 最小サブセット（prometheus.scrape, prometheus.remote_write, loki.write, otelcol.receiver.otlp, discovery.kubernetes）のlinux/amd64 Dockerビルド

## 実装体制（役割分担）

各マイルストーン内で、細かな実装の範囲は **Codex CLI（codex:rescue サブエージェント経由）または Cursor CLI** に委譲し、**Claude Code（Fable 5）は計画・タスク仕様の作成・実装結果の検証**を担う。1チャンクのフロー:

1. Claude Codeがタスク仕様を書く（対象ファイル、受け入れ条件、参照すべき既存コード/スキーマを明記。仕様は `docs/plans/` 配下のマイルストーン計画に紐付け）
2. Codex CLI / Cursor CLI が実装
3. Claude Codeが検証: `code-review`（正当性）、`verify`（実動作の確認: テスト実行・`alloy validate`・ブラウザ操作・ビルド成果物の起動確認）、必要なら `difit` でユーザーレビューに回す
4. 検証で見つかった問題は修正タスクとして再委譲。マイルストーン完了条件を満たしたらコミット

エージェント間の作業引き渡しには **agmsg**（https://github.com/fujibee/agmsg — SQLite床のクロスエージェントメッセージング、MIT）を使う。タスク仕様の送付・完了通知・検証結果の差し戻しをagmsgメッセージで往復させる。導入はM0のセットアップに含める:

- 前提: `sqlite3` — **導入済み**（3.46.1、確認済み）
- Claude Code側: `/plugin install agmsg@fujibee-agmsg`（ランタイムは `~/.agents/skills/agmsg/` に入り、Codex/Cursor側からも共有される）— これがM0で残る唯一のagmsgセットアップ作業
- 配送モードは turn（ターン間inboxチェック）から開始し、必要ならmonitorに切替

## リスク

1. レジストリ走査APIが非公開 → 同一モジュール内ジェネレータならアクセサ追加可。M1冒頭で確認
2. all.go書換でコンパイル破壊（他所からimportされるパッケージ）→ `alwaysIncludePkgs` ＋ nightly smoke build
3. CGO/クロスコンパイル問題 → ビルドイメージのタグをバージョンごとにピン。darwinはネイティブのみ
4. リフレクションで写像不能な型 → `raw` kindで系統的に吸収。ジェネレータはハードフェイル禁止
5. AlloyバージョンごとのGoツールチェイン差（v1.17.1はgo 1.26.4）→ Docker戦略で隔離、ホスト戦略はプリフライトで明示エラー
6. otelcol Consumerの型付けが粗い → traces/metrics/logs分割capsule＋互換表で開始、v0.1では一部raw参照を許容
7. ライセンス: シリアライザ翻案元（alloy-configurator, Apache-2.0）のattributionを初日から

## 検証方法

- M1: 生成スキーマをspot-check＋schemas.yml再現性チェック
- M2: smoke-build成果物を実行し、含めたコンポーネントの設定が通り・除外したものが拒否されることを確認
- M3以降: golden `alloy validate` テストがシリアライザの正しさの常設証明
- M4/M6: ブラウザ実操作（Playwright/webapp-testingスキル利用可）でE2E確認
- 最終: macOS実機でホストネイティブdarwinビルドのデモ手順を通す
